# Personnel KPI field — design

**Date:** 2026-07-19
**Status:** Approved (brainstorming)

## Problem

A department's `overview.json` records `personnel` as `[{ role, duties }]`, synthesised by
the `summarize` agent in `data-repo`. Each role captures *what the person is responsible
for* (duties) but not *how their performance is measured*. We want to add a **KPI** field
per role — a list of Persian performance targets such as «برگزاری حداقل ۴ جلسه آموزشی در
ماه» — generated by the `summarize` agent and shown and editable in the UI Overview screen.

The KPIs largely live in each department's job-description **attachments** (e.g.
`dining/attachments/شرح شغل سرپرست سالن.docx`, already cached to `.text/*.txt` by the
`extract-attachment` CLI and already read by the agent in Step 2a). In the source docs the
targets are usually **woven into the duty sentences** rather than listed under an explicit
"KPI" heading, so the agent must recognise measurable targets and lift them out.

This change spans both repos: the `summarize` agent lives in `data-repo`; the JSON
contract, UI, and tests live in `code-repo`. It respects code/data separation (INV-2) —
the agent only synthesises data; the contract/UI only render and validate it.

## Decisions

- **Field name & shape:** `kpi` — an **array of Persian strings**, one string per KPI,
  mirroring `duties` exactly. No structured `{metric, target, period}` objects: they force
  brittle parsing and invite fabrication when a sub-field is absent.
- **Required, empty allowed:** `kpi` goes in the personnel item's `required` array; an
  empty array `[]` is valid (used when a role has no measurable target). No existing
  `overview.json` files exist, so there is no migration burden.
- **Sources:** all evidence — attachments (primary) + full transcripts + process records —
  the same set duties are drawn from. A performance target stated only in a meeting still
  counts.
- **KPI vs duty:** a *duty* is a responsibility; a *KPI* is a statement carrying a
  **measurable/verifiable target** (a number, percentage, deadline, or frequency). The two
  may derive from the same source sentence — the duty keeps the responsibility, the KPI
  captures the quantified target as a clean standalone string. Rephrasing to a crisp target
  is allowed; **inventing or altering any number/target is forbidden** (INV-3).
- **UI scope:** full display **and** edit, exactly like duties are edited today.

## Changes

### 1. Data contract — `schemas/overview.schema.json`

In `personnel.items.properties`, add alongside `duties`:

```json
"kpi": { "type": "array", "items": { "type": "string" } }
```

Add `"kpi"` to the personnel item's `required` array. `additionalProperties` stays
`false`, so this must land in the schema before any writer emits it.

### 2. Summarize agent — `data-repo/.claude/agents/summarize.md`

- **New Step 6b — Synthesise KPIs** (parallel to Step 6 duties). For each role, scan all
  evidence (attachments primary, plus transcripts + process records) for statements that
  carry a measurable/verifiable target — number, percentage, deadline, or frequency — and
  write each as a clean standalone Persian KPI string. Rephrasing to a crisp target is
  allowed; no number/target may be invented or altered (INV-3). A role with no such
  evidence gets `kpi: []`.
- **Merge behavior** (identical to duties): match by `role`; additively append new KPIs;
  never drop a prior KPI unless new evidence explicitly supersedes it.
- **Step 7 output object:** each personnel item now has exactly `role`, `duties`, `kpi`.
  Update the JSON example and the "no other keys" note.
- **Constraints table:** add a row — `kpi` present (array; may be `[]`), Persian, no
  personal names in KPI text (ARD §4.4), no fabricated numbers/targets (INV-3).

### 3. UI types — `ui/src/api/types.ts`

Change the personnel shape to `{ role: string; duties: string[]; kpi: string[] }`.

### 4. Overview screen — `ui/src/screens/Overview.tsx`

- **`Draft` type + `enter()`/`save()`:** the personnel draft gains `kpi: string[]`.
  `enter()` seeds it from `p.kpi` (`kpi: [...p.kpi]`); `save()` maps it through
  `kpi: p.kpi.map((x) => x.trim()).filter(Boolean)`, mirroring the existing duties lines.
- **Read view:** under each role, render its KPIs as a collapsible list **below the duties
  pills** (same `toFa(count)`-labeled, pill-style treatment as duties, e.g. a «… شاخص»
  count). Empty → a faint placeholder (e.g. «شاخصی ثبت نشده است.»), consistent with the
  description field's empty state.
- **Edit view:** add/remove/edit KPI lines using the same helpers pattern as
  `patchDuties`/`setDuty` (add a `patchKpi`/`setKpi` pair). An «افزودن شاخص» control adds a
  blank line; blanks are dropped on save by the `.filter(Boolean)` above.
- **Backend:** no change. `put_overview` already validates the request body against
  `overview.schema.json` and writes it verbatim, so `kpi` flows through automatically.

### 5. Tests / fixtures

- `tests/fixtures/overview.cooking.json` — add a `kpi` array to each personnel entry
  (shared by schema tests and backend `conftest.py`).
- `tests/test_overview_schema.py` — add cases: valid with `kpi`, valid with `kpi: []`,
  invalid when `kpi` is missing, invalid when `kpi` is not an array of strings.
- `ui/src/screens/Overview.test.tsx`, `Overview.edit.test.tsx`,
  `ui/src/api/hooks.write.test.tsx` — extend the inline personnel fixtures with `kpi`;
  assert KPIs render in the read view and that adding/removing a KPI round-trips through
  the PUT payload.

## Out of scope (YAGNI)

- Structured KPI objects (`{metric, target, period}`), numeric target extraction, or any
  KPI scoring/analytics — KPIs are stored and edited as plain Persian strings only.
- Other agents (`classify`, `extract`, `consolidate`) — unchanged.
- No "regenerate KPIs" button — agent re-runs refine them additively, exactly as they do
  for duties and descriptions.
