# Department description field — design

**Date:** 2026-07-16
**Status:** Approved (brainstorming)

## Problem

A department's `overview.json` currently records `sub_units` and `personnel` (roles +
duties), synthesised by the `summarize` agent in `data-repo`. There is no free-text
overview that describes the department as a whole. We want a short **one-to-two paragraph
Persian description** of each department, generated by the `summarize` agent and shown —
and editable — in the UI Overview screen.

This change spans both repos: the `summarize` agent lives in `data-repo`; the JSON
contract, UI, and tests live in `code-repo`. It respects the code/data separation
(INV-2) — the agent only synthesises data, the contract/UI only render and validate it.

## Decisions

- **Field name:** `description` (top-level string). Distinct from the existing
  `sub_units[].description` nested field.
- **Placement in UI:** a block directly under the department header, **above**
  «واحدهای زیرمجموعه», labeled **«معرفی دپارتمان»**.
- **Required, empty allowed:** `description` is in the schema `required` array; an empty
  string `""` is a valid value (used when evidence is too thin). No existing
  `overview.json` files exist, so there is no migration burden.

## Changes

### 1. Data contract — `schemas/overview.schema.json`

Add a top-level property alongside `name`:

```json
"description": { "type": "string" }
```

Add `"description"` to the `required` array. `additionalProperties` stays `false`, so
this must land in the schema before any writer emits it.

### 2. Summarize agent — `data-repo/.claude/agents/summarize.md`

- **New Step 6a — Synthesise department description.** From the same evidence set
  (process records + full transcripts + attachments), write one or two Persian paragraphs
  describing the department's purpose and scope. Same rules: no fabrication (INV-3),
  roles-not-names (ARD §4.4), Persian. If evidence is too thin, emit `""`.
- **Merge behavior** (mirrors sub-unit descriptions): preserve the existing `description`
  on re-runs; refine/rewrite only when new evidence is more complete or precise. Never
  blank a filled description merely because a later run lacked new evidence.
- **Step 7 output object:** add `"description"` between `name` and `sub_units`.
- **Constraints table:** add a row — description present (may be empty), Persian, no
  fabrication.

### 3. UI types — `ui/src/api/types.ts`

Add `description: string` to the `Overview` interface.

### 4. Overview screen — `ui/src/screens/Overview.tsx`

- **Read view:** new block under the header, above «واحدهای زیرمجموعه», labeled
  **«معرفی دپارتمان»**, rendering the paragraph(s) in a `Card` with relaxed leading.
  Empty → faint placeholder «شرحی ثبت نشده است.».
- **Edit view:** `Draft` type gains `description: string`; render a multi-row
  `<textarea>` using the same input styling as the sub-unit description field.
  `enter()` seeds it from `data.description`; `save()` includes
  `description: draft.description.trim()`.
- **Backend:** no change. `put_overview` already validates the request body against
  `overview.schema.json` and writes it, so the field flows through automatically.

### 5. Tests / fixtures

- `tests/fixtures/overview.cooking.json` — add a `description` value (shared by the
  schema tests and backend `conftest.py`).
- `tests/test_overview_schema.py` — add cases: valid with description, valid with `""`,
  invalid when `description` is missing.
- `ui/src/screens/Overview.test.tsx`, `Overview.edit.test.tsx`,
  `ui/src/api/hooks.write.test.tsx` — extend inline fixtures with `description`; assert
  it renders in the read view and round-trips through edit/save.

## Out of scope (YAGNI)

- No "regenerate from agent" button in the UI — agent re-runs refine it, exactly as they
  already refine sub-unit descriptions.
- No rich text / markdown — plain paragraphs, edited like every other field.
